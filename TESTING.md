# NineCell — подробный чеклист тестирования

Этот файл нужен, чтобы быстро проверить сервис перед защитой.

## 1. Запуск

```powershell
cd C:\Users\marin\PycharmProjects\AI_slop
docker compose down
docker compose up --build
```

Открыть:

- приложение: http://localhost:5000
- RabbitMQ: http://localhost:15672, логин `kanban`, пароль `kanban`

Демо-аккаунты:

- `admin / admin123`
- `user / user123`

## 2. Проверка хранения данных

1. Войдите как `admin`.
2. Создайте задачу кнопкой `+` справа снизу.
3. Обновите страницу.
4. Задача должна остаться.
5. Остановите контейнеры через `Ctrl + C`, затем снова запустите `docker compose up`.
6. Задача должна остаться, потому что она хранится в PostgreSQL volume.

Проверить в БД:

```powershell
docker compose exec postgres psql -U kanban -d kanban
```

```sql
SELECT id, title, priority, team_id, assignee_id, created_at FROM task ORDER BY id DESC;
\q
```

## 3. Проверка прав и команд

1. Войдите как `admin`.
2. Откройте `Команды`.
3. Создайте команду или используйте `Victory Group`.
4. Создайте invite-ссылку.
5. Скопируйте ссылку.
6. Откройте другой браузер / режим инкогнито.
7. Войдите как `user` или зарегистрируйте нового пользователя.
8. Перейдите по invite-ссылке.
9. Пользователь должен вступить в команду.
10. Задачи этой команды должны быть видны участникам команды.

## 4. Проверка real-time

1. Откройте два окна браузера.
2. В первом войдите как `admin`.
3. Во втором войдите как `user`.
4. Создайте задачу для команды `Victory Group` и назначьте `user`.
5. Во втором окне задача должна появиться без ручного обновления.
6. Перетащите карточку в другую колонку — второе окно тоже обновится.

## 5. Проверка очереди RabbitMQ

Во втором PowerShell:

```powershell
$body = @{
  source = "api"
  external_id = "demo-queue-001"
  title = "Срочно: баг оплаты"
  description = "Проверка RabbitMQ"
  tags = "bug, urgent, payment"
  team = "Victory Group"
  assignee = "user"
} | ConvertTo-Json

Invoke-RestMethod `
  -Uri "http://localhost:5000/api/incoming_tasks" `
  -Method POST `
  -Headers @{"X-API-Key"="dev-incoming-token"} `
  -ContentType "application/json" `
  -Body $body
```

Ожидаемый результат:

- API вернул `status: queued`.
- В RabbitMQ сообщение попало в очередь `incoming_tasks`.
- Worker обработал сообщение.
- На доске появилась задача.
- В таблице `incoming_task` статус стал `processed`.

Проверить:

```powershell
docker compose exec postgres psql -U kanban -d kanban
```

```sql
SELECT id, source, external_id, status, task_id, error FROM incoming_task ORDER BY id DESC LIMIT 10;
SELECT id, routing_key, status, attempts, error FROM outbox_message ORDER BY id DESC LIMIT 10;
SELECT id, type, task_id, created_at FROM event ORDER BY id DESC LIMIT 20;
\q
```

## 6. Проверка дедубликации

Повторите запрос из пункта 5 с тем же `external_id = demo-queue-001`.

Ожидаемо:

- новая карточка не создаётся повторно;
- входящая задача получает статус `duplicate` или API сообщает `duplicate`.

## 7. Проверка автоматизации

1. Войдите как `admin`.
2. Откройте `Правила`.
3. Создайте правило:
   - событие: `Когда задача создана`
   - поле: `priority`
   - оператор: `равно`
   - значение: `critical`
   - действие: `Переместить в колонку`
   - значение действия: `in_progress`
4. Создайте задачу с priority `critical`.
5. Она должна автоматически перейти в `In Progress`.

## 8. Проверка optimistic locking

1. Откройте одну и ту же задачу в двух браузерах.
2. В первом сохраните изменение.
3. Во втором попробуйте сохранить старую версию.
4. Backend должен вернуть `409 Conflict`, а UI покажет ошибку о том, что задача уже изменена.

## 9. Проверка Telegram

1. Создайте бота через `@BotFather`.
2. В `.env` укажите:

```env
TELEGRAM_BOT_TOKEN=ваш_токен
TELEGRAM_BOT_USERNAME=имя_бота_без_@
```

3. Перезапустите:

```powershell
docker compose down

```

4. В Telegram напишите боту:

```text

```

5. Введите код на странице входа.

6. Для создания задачи из Telegram напишите:

```text

```

Ожидаемо: задача появится на доске через RabbitMQ и worker.

## 10. Что показать жюри

1. Два браузера, real-time обновление.
2. Invite-ссылка в команду.
3. Входящая задача через API → RabbitMQ → worker → карточка.
4. RabbitMQ UI: queues `incoming_tasks`, `automation_events`, dead-letter queues.
5. PostgreSQL таблицы: `task`, `incoming_task`, `outbox_message`, `event`.
6. Правило автоматизации.
7. Telegram-вход или Telegram-задача.

## Проверка команд, приглашений и видимости задач

### 1. Проверить управление командой
1. Войдите как `admin / admin123`.
2. Откройте левое меню → **Команды**.
3. Создайте команду, например `QA Team`.
4. В карточке команды нажмите **Редактировать** и измените описание.
5. В блоке **Добавить уже зарегистрированного пользователя** выберите `user`, роль `member`, нажмите **Добавить**.
6. Проверьте, что `user` появился в списке участников.
7. Нажмите **Сделать admin/member**, чтобы проверить смену роли.
8. Нажмите **Удалить** у участника, чтобы проверить удаление из команды.

### 2. Проверить invite-ссылку
1. Войдите как `admin`.
2. Откройте **Команды**.
3. В нужной команде выберите роль для приглашения, например `member`.
4. Нажмите **Создать ссылку**.
5. Ссылка автоматически скопируется. Также она появится в карточке команды.
6. Откройте другой браузер или окно инкогнито.
7. Вставьте ссылку.
8. Если вы не вошли в аккаунт, сначала войдите или зарегистрируйтесь.
9. После перехода по ссылке пользователь вступит в команду.

Важно: если проект запущен на `localhost`, invite-ссылка работает только на этом компьютере. Чтобы отправить ссылку другому человеку в Telegram, сервис должен быть запущен на сервере или публичном адресе.

### 3. Проверить, кто видит задачи
1. Войдите как `admin` в одном браузере.
2. Войдите как `user` в другом браузере или инкогнито.
3. Убедитесь, что оба пользователя состоят в одной команде.
4. Как `admin` создайте задачу и выберите эту команду.
5. У `user` задача должна появиться без перезагрузки.
6. Создайте задачу без команды. Её увидит только создатель, исполнитель и глобальный админ.

### 4. Проверить удаление команды
1. Войдите как админ команды.
2. Откройте **Команды**.
3. Нажмите **Удалить команду**.
4. Подтвердите.
5. Задачи не удаляются, но становятся без команды и без исполнителя.

## Prime v3 checklist

### 1. Проверить System Health

1. Войти как `admin`.
2. Открыть раздел **Статус**.
3. Нажать **Обновить**.
4. Проверить:
   - PostgreSQL доступен;
   - RabbitMQ доступен;
   - Redis доступен;
   - отображаются очереди `incoming_tasks`, `incoming_tasks_dead`, `automation_events`, `automation_events_dead`.

### 2. Проверить Demo Mode

1. Войти как `admin`.
2. Нажать **🚀 Demo Mode**.
3. Ожидаемо:
   - появляется команда `Prime Demo Squad`;
   - появляются демо-задачи;
   - создаётся входящая задача в очереди;
   - в событиях появляется `prime_demo_created`.

### 3. Проверить Retry входящей задачи

1. Открыть **Очередь**.
2. Найти запись со статусом `error`, `duplicate` или `processed`.
3. Нажать **Retry**.
4. Проверить, что статус стал `queued`, а затем worker обработал запись.

### 4. Проверить Activity Feed задачи

1. На карточке задачи нажать **🧾**.
2. Должно открыться окно истории задачи.
3. В нём должны отображаться события `task_created`, `task_updated`, `task_moved`, `automation_event_processed` и другие, если они были.

### 5. Проверить улучшенные фильтры

1. Открыть **Фильтры**.
2. Выбрать:
   - `Только мои`;
   - `Без исполнителя`;
   - `Просроченные`;
   - `Дедлайн сегодня`.
3. Проверить, что доска меняется без удаления данных.
